# 终端智能体前端选型调研报告

> 调研日期:2026-07-19。106 个并行调研代理、504 次检索,关键结论经 3 票对抗验证。
> 问题:为本地编码 AI(mlx_lm.server @ :7070 + Qwen3-Coder-30B-A3B + moon RAG)选一个类 Claude Code 的终端智能体前端——套壳现成开源 CLI,还是自研?

## 一、结论(TL;DR)

**套壳,不自研。首选 Qwen Code,备选 OpenCode。** RAG 用 FastMCP 包一层接入(几十行代码)。推理服务先实测现有 mlx_lm.server 的工具调用,不达标就换 mlx-openai-server(继续用同一份 MLX 权重,不用换模型格式)。

## 二、推荐方案

### 首选前端:Qwen Code(阿里官方,Apache-2.0,~26k stars)
- 就是为我们正在用的 Qwen3-Coder 模型调优的官方 CLI(Gemini CLI 改造),对话/读写文件/执行命令/权限确认等能力齐全。
- 接本地端点是**官方文档一等路径**:环境变量 `OPENAI_BASE_URL=http://localhost:7070/v1` + `OPENAI_MODEL=...`,或 settings.json 的 `modelProviders.baseUrl`;文档明确给了本地推理服务(vLLM/Ollama/LM Studio)的配置示例。
- **原生支持 MCP**(settings.json `mcpServers`,支持 stdio/SSE/HTTP 三种传输)→ 我们的 `moon` RAG 检索可以作为工具接入。
- 两个已知坑:本地端点也必须填一个非空占位 API key;上下文建议 ≥16k,太小模型会"忘掉"工具。

### 备选前端:OpenCode(MIT,~18.7 万 stars,已迁移到 anomalyco 组织)
- 最大的开源终端编码智能体,75+ provider;mlx-lm 维护者 awni 本人发过针对本地 MLX server 的完整 opencode.json 配置示例。
- 同样完整支持 MCP(stdio + remote)。
- 若 Qwen Code 体验不佳,平移成本很低。

### RAG 集成:FastMCP 包装 moon(标准做法)
- MCP 是接入这些 CLI 的标准协议;FastMCP(PrefectHQ,Apache-2.0,26.3k stars,3.0 已发布)是事实标准框架。
- moon 本身是 Python 包,`@mcp.tool` 装饰检索函数 + `mcp.run()` 即可,预计几十行。
- 注意:MCP 工具描述会占上下文 token,本地模型上下文有限,工具数量宜精不宜多。

### 推理服务:先实测,必要时替换
- **mlx_lm.server 的工具调用历史上"基本是坏的"**(维护者 awni 原话,2026-01 PR #711 修复前);修复后**仍"仅限部分模型"**。好消息:解析器自动检测恰好匹配 Qwen3-Coder 的模板标记(issue #1293),我们的模型是相对有利的情况——但必须实测确认。
- 不达标则换 **mlx-openai-server**(cubist38,FastAPI,活跃维护,v1.8.1):直接加载同一份 mlx-community 权重,自带 `--tool-call-parser qwen3_coder` 专用解析器。注意这是单一来源信息且有零星失败报告(issue #181),定位是"值得实测的候选"而非"已证明更好"。

## 三、为什么不选其他路线

### Claude Code 本体连本地模型:可行但绕路,不推荐
- Claude Code 是 source-available(非开源),`ANTHROPIC_BASE_URL` 指向的端点必须讲 **Anthropic Messages API**,裸 OpenAI 兼容服务不能直连。
- 路径 A:LiteLLM 代理做协议翻译——成熟但有边角问题(>64 字符工具名、参数兼容),且 LiteLLM 有过 PyPI 版本被投毒的供应链安全记录(1.82.7/1.82.8)。
- 路径 B:Ollama 0.14+ 原生实现了 Anthropic API,可免代理直连——但要放弃 MLX 权重改用 GGUF,推理栈整个换掉。
- Anthropic 官方不支持把 Claude Code 路由到非 Claude 模型。

### 自研:可行但性价比低
- OpenAI Agents SDK + 自写 shell/文件工具约 300-400 行可跑通,但权限系统、diff 展示、会话管理、MCP 这些成熟 CLI 都是现成的,自研全要重做。定位为兜底/学习项目。

## 四、落地步骤(建议顺序)

1. `npm i -g @qwen-code/qwen-code`,配 `OPENAI_BASE_URL=http://localhost:7070/v1`(key 填占位值)
2. 实测工具调用:让它读一个文件、跑一条命令,观察是否正确解析(这是 go/no-go 判据)
3. 若解析失败 → 换 mlx-openai-server(同一份权重)再测
4. 用 FastMCP 写 `moon mcp` 子命令,把 `moon search`(检索)暴露为 MCP 工具,配进 Qwen Code 的 `mcpServers`
5. serve.sh 相应调整(端口不变 7070)

## 五、证据缺口(诚实标注)

1. Crush、Aider、Goose、Codex CLI、Gemini CLI 没有任何结论通过验证——对比只覆盖 Qwen Code / OpenCode / Claude Code 三者。
2. "Qwen3-Coder 30B 配各 CLI 的社区实测成功率"几乎无幸存结论;"哪个推理后端对该模型工具调用最好"未被证实,推荐是基于间接证据的工程判断,**以我们自己的实测为准**。
3. mlx-openai-server 的解析器细节是单一来源,置信中等。
4. 36GB 内存下 30B + ≥16k 上下文 + MCP 工具描述的实际余量未实测(KV cache 会涨)。
5. 时效提醒:OpenCode 组织已迁移(sst→anomalyco),配置细节以当期官方文档为准。
