# 个人编码 AI 技术方案调研报告

> 调研日期:2026-07-19。由 105 个并行调研代理完成,537 次检索,每条关键结论经 3 票对抗验证(≥2 票否决即剔除)。
> 目标:在 MacBook Pro M5 Max(36GB 统一内存)上,基于开源模型打造一个强编码能力、且学习个人代码与文档的 AI。

## 一、核心结论(TL;DR)

1. **基座模型选 Qwen3-Coder-30B-A3B-Instruct**:MoE 架构,总参 30.5B、每 token 只激活 3.3B,推理速度接近 3B 小模型但能力是 30B 级;Apache 2.0 开源,原生 256K 上下文。4-bit 量化 18.6GB、6-bit 24.8GB,36GB 内存可稳定本地推理(8-bit 32.5GB 装不下,macOS 默认 GPU 可用约 27GB)。
2. **先建 RAG,不要急着微调**。工业级实证(腾讯 WXG,16 万+文件代码库,FSE 2025)显示:检索增强的性能上限**高于**微调(精确匹配 53.76% vs 44.20%),且朴素的 BM25 词法检索甚至优于神经向量检索。
3. **微调是第二步优化**:RAG + 微调结合比单用任一方案平均再提升约 7.8 个点(EM)。微调用 **LoRA/QLoRA**——已证实比全参微调更能保住基座模型的通用能力(缓解灾难性遗忘)。全参微调本地完全不可行(6.7B 模型就需要 67GB+ 显存)。
4. **小项目可以不用 RAG**:约 4 万 token 以内的结构良好代码库,直接全部塞进 256K 长上下文,效果持平甚至更好。RAG 的价值随代码库变大才显现。
5. **微调有代价**:域内提升的同时,通用编码基准会明显下降(HumanEval 平均 -24.1% Pass@1)。所以微调模型用于"你的项目",通用任务保留原始基座——两者可通过 LoRA 适配器热切换,成本几乎为零。

## 二、推荐方案(分三阶段)

### 阶段 1:本地推理跑通(第 1 周)
- 下载 Qwen3-Coder-30B-A3B-Instruct 的 MLX 4-bit 量化版,本地起 OpenAI 兼容服务。
- 小项目场景直接用长上下文喂入全部代码,立即可用。
- 产出:一个能日常使用的本地编码助手。

### 阶段 2:RAG 管线(第 2~3 周)
- **分块**:AST 结构感知分块(cAST 方法,CMU 2025,已开源 `astchunk` 工具包)。实证:比固定大小分块 Recall@5 +4.3、下游 Pass@1 平均 +5.5。绝不用按行/固定字符切分(会切散函数和类)。
- **检索**:BM25 为主(工业实证最强)+ 向量检索混合。
- **组件选型**(全本地):Qwen3-Embedding-4B(2560 维,可 MRL 降维)+ Qwen3-Reranker-4B + LanceDB(嵌入式向量库,无需服务)。参考项目 QwenRag(注意:该项目很小,仅作选型示例,管线自己写)。
- 产出:AI 能准确引用你的代码库和文档回答问题。

### 阶段 3:LoRA 微调(第 4 周起,可选)
- 目的:学你的代码风格、命名习惯、项目"方言"——这是检索给不了的。
- 先用 7B 级模型(如 Qwen 系 coder 7B)在本地 MLX 上跑通 LoRA 管线、验证数据质量,再决定是否对 30B 上云训练。
- **注意**:30B QLoRA 能否装进 36GB 本地内存,调研未找到可验证的数据(见证据缺口),需实测;不行就上云。
- 云端价格(2026-07 快照):RTX 4090 约 $0.14~0.34/小时(够 7B~13B 微调);A100 80GB $0.68(spot)~4.10(按需)/小时(13B~70B)。
- 产出:一个带你个人风格的 LoRA 适配器,可与基座热切换。

## 三、关键实证依据(均 3-0 通过对抗验证,另注明的除外)

| 结论 | 数据 | 来源 |
|---|---|---|
| RAG 上限高于微调 | BM25 53.76% vs 微调 44.20% EM | 腾讯 FSE 2025(arxiv 2505.15179) |
| RAG+微调更优 | EM +7.79 / 编辑相似度 +5.27 / BLEU +7.91 | 同上 |
| 全参微调不可行 | 6.7B 全参需 67.2GB 显存、41.4 小时(8×A100) | 同上 |
| LoRA 缓解遗忘 | 优于权重衰减/dropout;"learns less and forgets less" | Biderman et al., TMLR 2024 |
| 长上下文 vs RAG 边界 | ~40k token 以下长上下文持平或更优 | RACG 综述(arxiv 2510.04905,2026-05 更新) |
| AST 分块有效 | Recall@5 +4.3(最佳检索器;跨检索器平均 +2.8)、Pass@1 +5.5 | cAST(arxiv 2506.15655) |
| Qwen3-Coder-30B 内存 | Q4_K_M 18.6GB / MLX 6-bit 24.81GB / 8-bit 32.46GB | HF 官方量化仓库逐字节核验 |
| 量化内存经验法则 | Q4_K_M ≈ 参数量(B)×0.6GB + 2~4GB 开销(实测误差<1%) | 经真实 GGUF 文件交叉验证 |
| 云 GPU 价格 | 4090:$0.14~0.34/hr(2-1 通过,下限存疑);A100 80GB:$0.68~4.10/hr | Lambda/RunPod 官方定价页 |

## 四、证据缺口与风险(需实验验证,不要当作已知)

1. **MLX 在 M5 Max 上的实际训练速度**:没有任何社区数据通过验证(M5 太新)。阶段 3 前先跑小规模基准。推理速度参考:M4 Max 上 Qwen3-30B-A3B 4-bit 实测约 70~110 tok/s(Apple 官方 mlx-lm 基准 113 tok/s,社区实测 87 tok/s@8K 上下文,数字有分歧)。
2. **30B QLoRA 的本地可行性未决**:"30B LoRA 需 80GB+"的说法被否决(没算量化微调),但也没有正面证据证明 36GB 装得下。
3. **微调数据怎么从个人代码库生成**(指令对、FIM、continued pretraining 的具体方法):无通过验证的方法学结论,属于开放问题,阶段 3 需要自己实验。
4. **IDE 接入**(Continue.dev、Zed、Cline 对本地模型的支持细节):未获验证数据,待实际配置时确认。
5. 腾讯论文的场景是单一 C++ 大代码库、≤7B 模型、行级补全,外推到个人多语言项目 + 30B 模型需保留不确定性。
6. macOS GPU 内存上限默认约为总内存 75%(可用 sysctl 调高),"装得下"的边界有弹性;超长上下文时 KV cache 会显著超出 2~4GB 的开销估算。
7. GPU 现货价格波动大,上表仅为 2026 年 7 月快照。

## 五、备选方案

- **基座备选**:需要更强推理能力时,DeepSeek-R1 官方蒸馏版(1.5B~70B)中 32B 及以下 4-bit 可在 36GB 上运行。密集模型备选 Qwen2.5-Coder-32B(Q4_K_M 约 19.9GB),但推理速度远慢于 MoE 的 30B-A3B。
- **硬件不够时**:全程在云端 4090/A100 上训练,本地只做推理和 RAG。
- **最简路线**:如果代码库不大,阶段 1(长上下文)+ 简化版阶段 2 就足够,可能根本不需要微调。
